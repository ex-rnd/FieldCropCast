## 1. Overview

FieldCropCast is a real-time weather intelligence dashboard built for Kenyan farmers. It combines live weather data, AI-generated field summaries, crop-specific risk assessment, satellite tree analysis, and M-Pesa subscription payments into a single responsive interface.

**Primary users:** Smallholder and commercial farmers across Kenya's 47 counties.

**Core value proposition:**
- Translate raw meteorological data into plain-language farming actions
- Surface crop-specific risk levels (rain, wind, heat/frost) for the next 7 days
- Identify tree health from field photographs using satellite-grade AI
- Persist farm profiles across devices via Firebase Firestore

## 2. Architecture

```
Browser (Client Components)
  │
  ├── Clerk (Authentication)
  ├── localStorage  (fc-farm, fc-theme — fast boot cache)
  └── Firebase Firestore  (cloud profile sync)
        │
        ▼
Next.js App Router (Server)
  │
  └── /app/backend/api/*  (proxy route handlers)
        │
        ├── WeatherAI API  (weather, AI summary, tree analysis)
        └── Safaricom Daraja API  (M-Pesa STK Push)
```

The application is entirely client-rendered (`'use client'`). All external API calls are proxied through Next.js Route Handlers to keep credentials server-side. The frontend never holds API keys.

## 3. Features

### 3.1 Farm Setup (Left Sidebar)

| Field | Details |
|---|---|
| Farm Name | Free text, persisted to localStorage and Firestore |
| County / Region | Dropdown of all 47 Kenya counties with automatic lat/lon population |
| Location | Auto-populated from county selection; also accepts browser geolocation |
| Crop | Chosen in the Center Panel crop selector |
| Units | Metric (°C, mm, km/h) or Imperial (°F, in, mph) |

**Auto-geolocation:** On first load the browser Geolocation API is queried. If granted, latitude and longitude are populated automatically without requiring county selection.

**AI Usage Quota:** Displays current period request count, AI request count, and remaining quota against plan limits with a visual progress bar.

### 3.2 Weather Intelligence (Center Panel)

#### Crop Selector
Supports 17 crop types: General, Maize, Tea, Coffee, Wheat, Rice, Beans, Tomatoes, Potatoes, Sugarcane, Flowers, Bananas, Green Grams, Cowpeas, Sorghum, Cassava, Mangoes.

Risk thresholds and field recommendations are recalculated immediately on crop change without a network round-trip.

#### Field Recommendations
Derived from the 7-day forecast using crop-specific risk thresholds. Each recommendation carries one of four urgency levels:

| Level | Trigger |
|---|---|
| `urgent` | Critical risk — immediate action required |
| `warning` | High risk — action recommended within 24 hours |
| `caution` | Moderate risk — monitor closely |
| `good` | Low risk — conditions favourable |

#### Crop Details
Optional inputs that refine recommendations:
- **Planting Date** — enables crop growth stage estimation
- **Farm Size** — `< 1 acre` to `> 10 acres` (used for tree analysis context)
- **Water Source** — Rain-fed or Irrigated (adjusts irrigation recommendations)
- **Soil Type** — Sandy, Loam, Clay, Silt, Peat, Chalk (affects drainage advice)
- **Crop Variety** — Variety-specific sub-selection per crop type

#### Estimated Crop Stage
When a planting date is provided, the system estimates the current growth stage (e.g. Germination → Vegetative → Flowering → Maturity) with a progress bar and stage-specific agronomic advice.

#### Current Conditions
Six real-time metrics from the current hour's forecast:
Rain Today · Feels Like · UV Index · Rain Chance · Humidity · Wind Speed & Direction

#### 7-Day Risk Assessment
Three risk cards (Rain · Wind · Heat/Frost) each displaying:
- Risk level (Low / Moderate / High / Critical) with colour coding
- Scored 0–100 with a filled progress bar
- Detail string (e.g. peak precipitation, max wind speed, high/low temperatures)

### 3.3 AI Summary & Field Photos (Right Sidebar)

#### AI-Generated Summary
Narrative weather summary generated by the WeatherAI API. Available in two languages:
- **SW** — Kiswahili (default, fetched with initial weather call)
- **EN** — English (fetched on demand via a second API call)

Falls back to a locally-generated summary if the API does not return one.

#### Field Photos — Tree Analysis
Upload a field photograph (JPG, PNG, WEBP, up to 20 MB) to run satellite-grade tree analysis via the WeatherAI Trees API.

**Response data displayed:**
- Annotated overlay image (linked, opens in new tab)
- Tree count · Density per acre · Canopy coverage % · Confidence score
- Health breakdown: Healthy / Needs Care / Needs Replacement (tri-colour bar)
- Detected species
- AI observations and recommendations

**Tree Analysis Quota:** Monthly usage bar showing analyses used vs. limit with reset date.

**Analysis History:** Collapsible list of up to 10 previous analyses with date, tree count, canopy coverage, and link to annotated image.

## 4. Data Model

### FarmState (localStorage key: `fc-farm`)

```typescript
interface FarmState {
  name: string;           // Farm / farmer name
  county: string;         // Kenya county
  lat: string;            // Decimal latitude
  lon: string;            // Decimal longitude
  crop: string;           // Crop type slug
  units: 'metric' | 'imperial';
  plantingDate?: string;  // ISO date YYYY-MM-DD
  farmSize?: string;      // '< 1 acre' | '1–2 acres' | ...
  irrigationType?: 'rain_fed' | 'irrigated';
  soilType?: string;
  cropVariety?: string;
}
```

### Persistence priority (boot order)
1. `localStorage` — loaded synchronously on mount (instant UI)
2. Firebase Firestore (`farmers/<email>`) — loaded async; cloud record wins if it exists and overwrites localStorage

### Write triggers
`FarmState` is written to both localStorage and Firestore (fire-and-forget) on:
- Clicking **Analyze Farm**
- Changing the selected crop
- Any Crop Details field change

## 5. API Reference

All endpoints are Next.js Route Handlers under `/app/backend/api/`. They proxy to WeatherAI and Safaricom — credentials are never exposed to the client.

### Error response shape

All error responses follow a consistent structure:

```json
{
  "code": "QUOTA_EXCEEDED | AUTH_ERROR | UPSTREAM_ERROR | TIMEOUT",
  "error": "Human-readable message",
  "detail": "Raw upstream message",
  "resetAt": "2025-08-01T00:00:00Z"
}
```

| Code | HTTP Status | Cause |
|---|---|---|
| `QUOTA_EXCEEDED` | 429 | WeatherAI monthly quota exhausted |
| `AUTH_ERROR` | 401 | Invalid or expired WAI_API_KEY |
| `UPSTREAM_ERROR` | 502 | WeatherAI 5xx or malformed response |
| `TIMEOUT` | 504 | Fetch exceeded timeout (15s weather, 30s trees) |

---

### `GET /backend/api/weather`

Proxies WeatherAI `/v1/weather`. Returns current conditions, 7-day daily forecast, hourly forecast, and AI summary.

**Query parameters:**

| Param | Required | Default |
|---|---|---|
| `lat` | ✓ | — |
| `lon` | ✓ | — |
| `days` | | `7` |
| `units` | | `metric` |
| `ai` | | `true` |
| `lang` | | `sw` |

---

### `POST /backend/api/trees/analyze`

Proxies WeatherAI `/v1/trees/analyze`. Accepts `multipart/form-data`.

**Form fields:** `image` (file), `farmerId`, `county`, `landAcres`, `notes`, `location`

**Response:** Annotated overlay image URL, tree count, density, canopy coverage, confidence score, health breakdown, species, observations, recommendations.

---

### `GET /backend/api/trees/quota`

Returns current period tree analysis usage and limit.

### `GET /backend/api/trees/history`

Returns list of previous tree analyses for the API key.

### `GET /backend/api/usage`

Returns WeatherAI plan, request counts, AI request counts, and limits for the current period.

### `GET /backend/api/config`

Returns static UI configuration: crop list, unit options, auto-refresh interval, version string.

### `GET /backend/api/geocode`

Reverse geocodes a lat/lon pair to a place name.

### `GET /backend/api/alerts` · `DELETE /backend/api/alerts`

GET: Returns active weather alerts.  
DELETE with `?id=`: Dismisses a single alert.  
DELETE without params: Dismisses all alerts.

### `POST /backend/api/alerts/webhook`

Receives incoming WeatherAI webhook payloads. Optionally validated against `WEBHOOK_SECRET` header.

### `POST /backend/api/mpesa/stk-push`

Initiates a Safaricom M-Pesa STK Push payment.

**Body:** `{ phoneNumber: string, amount: number, planId: string }`  
**Response:** `{ CheckoutRequestID, ResponseCode, CustomerMessage }`

### `GET /backend/api/mpesa/query`

Polls Safaricom for STK Push transaction status.

**Query:** `?checkoutId=<CheckoutRequestID>`  
**Response:** `{ ResultCode, ResultDesc }` — `ResultCode "0"` = success, `"1032"` = cancelled by user.

### `POST /backend/api/mpesa/callback`

Safaricom callback endpoint. Logs the transaction result and returns `{ ResultCode: 0, ResultDesc: "Accepted" }`.

## 6. Authentication

Authentication is handled by [Clerk](https://clerk.com).

- Sign-in and sign-up pages at `/sign-in` and `/sign-up` (Clerk hosted UI)
- All routes except `/sign-in` and `/sign-up` are protected via `middleware.ts`
- The farmer's primary email address is used as the Firestore document ID
- On sign-out: `fc-farm` and `fc-theme` are removed from localStorage before the Clerk redirect

## 7. Database & Persistence

### localStorage

| Key | Value | Purpose |
|---|---|---|
| `fc-farm` | `FarmState` JSON | Farm profile cache for instant boot |
| `fc-theme` | `"dark"` \| `"light"` | Theme preference |

### Firebase Firestore

- **Collection:** `farmers`
- **Document ID:** farmer's Clerk primary email address
- **Document shape:** `FarmState` object (same as localStorage)
- **Access pattern:** unauthenticated Firestore client (Clerk handles auth, not Firebase Auth)
- **Firestore rules** should restrict access to the `farmers` collection as appropriate for your security model

## 8. Payments

Payments use the **Safaricom Daraja API** (sandbox by default).

**Flow:**
1. Farmer selects a paid plan (Pro or Scale) in the Plans modal
2. `PaymentModal` opens — farmer enters their M-Pesa phone number
3. STK Push is initiated via `POST /backend/api/mpesa/stk-push`
4. Farmer receives a PIN prompt on their phone
5. Frontend polls `GET /backend/api/mpesa/query` every 8 seconds
6. On `ResultCode "0"`: success state shown
7. On `ResultCode "1032"` or 90-second timeout: failure state shown with retry option

**Note:** The `MPESA_CALLBACK_URL` must be a publicly reachable HTTPS URL. In development, use [ngrok](https://ngrok.com): `ngrok http 3000`.

## 11. Running Locally

### Prerequisites
- Node.js ≥ 18
- A WeatherAI API key
- A Clerk application
- A Firebase project with Firestore enabled
- Safaricom Daraja sandbox credentials

### Steps

```bash
# 1. Clone the repository
git clone https://github.com/ex-rnd/FieldCropCast.git
cd FieldCropCast

# 2. Install dependencies
npm install

# 3. Configure environment
cp .env.example .env
# Fill in all required variables (see Section 9)

# 4. Start ngrok for M-Pesa callbacks (development only)
ngrok http 3000
# Copy the HTTPS URL into MPESA_CALLBACK_URL in .env

# 5. Start the development server
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). Sign in via Clerk to access the dashboard.

### Type checking

```bash
npm run typecheck
```

## 12. Deployment

The application is deployed via [Coolify](https://coolify.io) using the Nixpacks build pack.

**Configuration:**
- **Repository:** `FieldCropCast` (GitHub)
- **Branch:** `main`
- **Base Directory:** `/`
- **Port:** `3000`
- **Build Pack:** Nixpacks

**Required environment variables:** All variables listed in [Section 9](#9-environment-variables) must be set in the Coolify environment configuration before deploying.

**M-Pesa callback URL:** Set `MPESA_CALLBACK_URL` to your production domain (e.g. `https://app.yourdomain.com/backend/api/mpesa/callback`) before going live. Replace the sandbox Daraja URLs with production endpoints when ready.
